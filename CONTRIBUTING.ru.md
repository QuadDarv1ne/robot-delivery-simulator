## Начало работы

### Требования

- Node.js 18+ (рекомендуется LTS)
- npm или bun
- Git
- Редактор кода (рекомендуется VS Code)

### Форк и клонирование

1. Сделайте форк репозитория на GitHub
2. Клонируйте свой форк локально:
   ```bash
   git clone https://github.com/YOUR_USERNAME/robot-delivery-simulator.git
   cd robot-delivery-simulator
   ```
3. Добавьте upstream remote:
   ```bash
   git remote add upstream https://github.com/original/robot-delivery-simulator.git
   ```

## Как внести вклад

### Сообщение об ошибках

Перед созданием отчета об ошибке проверьте существующие issues. При создании отчета включите как можно больше деталей:

- Используйте четкий и описательный заголовок
- Опишите точные шаги для воспроизведения проблемы
- Приведите конкретные примеры
- Опишите наблюдаемое и ожидаемое поведение
- Включите скриншоты, если полезно
- Укажите ваше окружение (ОС, браузер, версия Node)

### Предложение улучшений

Предложения по улучшению отслеживаются как GitHub issues. При создании предложения:

- Используйте четкий и описательный заголовок
- Предоставьте подробное описание предлагаемого улучшения
- Объясните, почему это улучшение было бы полезным
- Перечислите любые альтернативы, которые вы рассматривали
- Включите макеты или примеры, если применимо

### Ваш первый вклад

Не знаете, с чего начать? Ищите issues с метками:
- `good first issue` — хорошо для новичков
- `help wanted` — требуется дополнительное внимание
- `documentation` — улучшения документации

## Настройка окружения разработки

### Установка

```bash
# Установить зависимости
npm install

# Настроить базу данных
npx prisma generate
npx prisma db push

# Создать демо-пользователя
npm run seed

# Запустить серверы разработки
npm run dev
```

```bash

# E2E тесты
```

## Стандарты кодирования

### TypeScript

- Используйте TypeScript для всех новых файлов
- Определяйте правильные типы, избегайте `any`
- Используйте интерфейсы для форм объектов
- Используйте псевдонимы типов для объединений/примитивов

### React

- Используйте функциональные компоненты с хуками
- Следуйте правилам хуков
- Держите компоненты небольшими и сфокусированными

- Мы используем ESLint и Prettier

- Форматируйте код с помощью `npm run format`
- Максимальная длина строки: 100 символов

## Правила коммитов

Мы следуем спецификации [Conventional Commits](https://www.conventionalcommits.org/).

### Формат сообщения коммита

```
<тип>(<область>): <описание>

[необязательное тело]

[необязательный футер]
```

### Типы

- `feat`: Новая функция
- `fix`: Исправление ошибки
- `docs`: Только изменения документации
- `style`: Изменения, не влияющие на смысл кода
- `refactor`: Изменение кода без исправления ошибок или добавления функций
- `perf`: Изменение кода, улучшающее производительность
- `test`: Добавление или исправление тестов
- `chore`: Изменения в процессе сборки или вспомогательных инструментах

### Примеры

```
feat(map): add 3D terrain visualization
fix(auth): resolve session timeout issue
docs(readme): update installation instructions
refactor(api): restructure user endpoints
```

## Процесс Pull Request

### Перед отправкой

1. Обновите свою ветку до последней версии upstream:
   ```bash
   git fetch upstream
   git rebase upstream/main
   ```

2. Запустите все тесты:
   ```bash
   npm run test
   npm run lint
   npm run build
   ```

3. Обновите документацию при необходимости

### Отправка

1. Отправьте в свой форк:
   ```bash
   git push origin your-feature-branch
   ```

2. Откройте Pull Request против `main`

3. Полностью заполните шаблон PR

4. Ссылайтесь на связанные issues

### Процесс рецензирования

- PR требует минимум одно одобрение
- Все CI проверки должны пройти
- Оперативно реагируйте на отзывы рецензентов
- Держите PR актуальным с main

### После слияния

- Удалите свою ветку функции
- Обновите локальную ветку main
- Празднуйте! 🎉
